# Observability & Telemetry

AION Agent V3 integrates a complete native observability stack, compatible with OpenTelemetry standards, designed for advanced monitoring of LLM agents in production environments.

## Architecture Overview

AION's observability is structured around **three pillars** — Traces, Metrics, Logs — all based on a **push-based** model towards an OTel collector (`AION_OTEL_ENDPOINT`). In addition, there is a second telemetry layer dedicated solely to the LLM component: **Opik** (Comet ML), a self-hosted system for prompt library management, LLM call tracing and session feedback scores.

```
┌─────────────────────────────────────────────────────────────┐
│                    AION Agent Runtime                        │
│                                                             │

│  │ (OTel)   │   │ (Dual)   │   │(struct)  │               │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘               │
│       │               │              │                      │
│  ┌────▼─────┐   ┌────▼─────┐   ┌────▼─────┐               │

│  └────┬─────┘   └────┬─────┘   └────┬─────┘               │
```

### Main Components

| Pillar | Technologies | Mode | Endpoint / Destination |
|---|---|---|---|
| **Traces** | OpenLit → OTel SDK | OTLP Push | `AION_OTEL_ENDPOINT` |
| **Metrics** | Prometheus + OTel | Dual-export (Push + Pull) | `/metrics` (local) + `AION_OTEL_ENDPOINT` |
| **Logs** | structlog + OTel | OTLP Push | `AION_OTEL_ENDPOINT` |
| **LLM Telemetry** | Opik (Comet ML) | HTTP Push | `OPIK_URL_OVERRIDE` |

## Tracing (OpenTelemetry)

AION emits hierarchical spans for each critical processing phase. Traces use `aion.*` attributes (not `gen_ai.*`) and are compatible with any backend that accepts OTLP (Jaeger, Tempo, etc.).

### OpenLit vs Standard OTel SDK

AION first attempts to initialize **OpenLit** (full auto-instrumentation: FastAPI, SQLAlchemy, HTTPX). If OpenLit is unavailable or fails, it falls back to the **standard OTel SDK fallback**.

The init flow is located in `src/observability/otel_setup.py`:

1. If `AION_OTEL_ENABLED=1` and `AION_OPENLIT_ACTIVE=1` → uses OpenLit (already initialized in `aion_env.py`)
2. Otherwise if `AION_OTEL_ENABLED=1` → fallback to standard OTel SDK with `BatchSpanProcessor`
3. If `AION_OTEL_ENABLED=0` → local dummy tracer provider (no export)

### Span Flow

Each conversational turn generates a tree similar to the following:

```
conversation.turn (root)
├── hooks.pre_llm_call
├── llm.call (TTFT, Total Latency)
├── tool.execute (e.g. prometheus_query)
│   ├── mcp.request (cross-process propagation)
└── hooks.post_llm_call
```

### Span Attributes

Spans use the `aion.*` prefix for custom metadata:

| Attribute | Description |
|---|---|
| `aion.profile` | Name of the active agent profile |
| `aion.session_id` | Chat session ID |
| `aion.user_id` | User ID |
| `aion.tenant_id` | Tenant ID |
| `tool.name` | Name of the executed tool |
| `tool.mcp_server` | Name of the MCP server |
| `tool.output` | Tool output (if brief) |
| `tool.error` | Error message |

### Trace Correlation

The `trace_id` generated by OpenTelemetry is automatically propagated to the persistence layer and saved within the unified DB in the following tables:

* `messages`
* `steps` (tool calls)
* `audit_log`

This allows starting from an application log or an audit and pivoting to the trace visualization in a compatible OTel backend (Jaeger, Tempo, Phoenix, etc.).

### Tracing Configuration

| Variable | Default | Description |
|---|---|---|
| `AION_OTEL_ENABLED` | `0` | Enables OTel tracing (0 = disabled, local tracer only) |
| `AION_OTEL_ENDPOINT` | `http://localhost:4317` | OTLP collector endpoint |
| `AION_OTEL_PROTOCOL` | `grpc` | OTLP protocol: `grpc` or `http` |
| `AION_OTEL_SERVICE_NAME` | `aion-agent` | Service name in traces |
| `AION_OTEL_TRACE_CONTENT` | `1` | Captures LLM content in spans (prompt/response) |

## Metrics

AION uses a **dual-export pattern**: each metric is simultaneously backed by Prometheus (`prometheus_client`) and OTel. The wrappers in `src/observability/metrics.py` (`OTelPrometheusCounter`, `OTelPrometheusHistogram`, `OTelPrometheusGauge`) ensure that every `inc()`, `observe()` or `set()` operation writes to both backends.

### Prometheus Exposure

Metrics are exposed locally on the `/metrics` endpoint (default) via `prometheus_fastapi_instrumentator`, enabled by `AION_METRICS_ENABLED=1`. This allows Prometheus/Grafana to scrape directly without additional configuration.

### OTel Metrics Push

When `AION_OTEL_ENABLED=1`, metrics are also periodically pushed to `AION_OTEL_ENDPOINT` via `PeriodicExportingMetricReader` with a configurable interval via `AION_OTEL_METRIC_EXPORT_INTERVAL` (default 5000ms).

### Wrapper Classes

Each metric is a wrapper that maintains a native Prometheus metric and an OTel instrument:

| Wrapper | Prometheus | OTel |
|---|---|---|
| `OTelPrometheusCounter` | `Counter` | `Counter` |
| `OTelPrometheusHistogram` | `Histogram` | `Histogram` |
| `OTelPrometheusGauge` | `Gauge` | `Gauge` (or `UpDownCounter` as fallback) |

`_init_otel()` is called after the `MeterProvider` has been configured (in `otel_setup.py`), linking the OTel instruments to the wrappers.

### Metrics Inventory

AION defines **11 metrics** divided into 3 categories:

#### Conversational Metrics

| Metric | Type | Labels | Description |
|---|---|---|---|
| `aion_messages_total` | Counter | `tenant_id`, `profile`, `role`, `finish_reason` | Total messages processed (user or assistant) |
| `aion_tool_calls_total` | Counter | `tenant_id`, `profile`, `tool_name`, `mcp_server`, `status` | Total tool invocations (status: ok, error, blocked) |

#### Performance Metrics

| Metric | Type | Labels | Description |
|---|---|---|---|
| `aion_turn_duration_seconds` | Histogram | `tenant_id`, `profile` | End-to-end turn duration (buckets: 0.5–120s) |
| `aion_tool_call_duration_seconds` | Histogram | `tool_name`, `mcp_server` | Single tool execution duration (buckets: 0.05–30s) |
| `aion_session_cache_size_bytes` | Gauge | `tenant_id` | Size of the session's local filesystem |

#### LLM Metrics (self-reported)

| Metric | Type | Labels | Description |
|---|---|---|---|
| `aion_llm_tokens_total` | Counter | `tenant_id`, `profile`, `model`, `token_type` | Cumulative tokens (token_type: prompt, completion, reasoning) |
| `aion_llm_turn_tokens` | Gauge | `tenant_id`, `profile`, `model`, `token_type` | Tokens of the last turn |
| `aion_llm_turn_calls` | Gauge | `tenant_id`, `profile` | Number of LLM calls in the last turn |
| `aion_agent_failures_total` | Counter | `tenant_id`, `profile`, `error_type` | Agent turn failures (error_type: timeout, error, cancelled, etc.) |
| `aion_mcp_server_healthy` | Gauge | `mcp_server` | MCP server health status (1=healthy, 0=unhealthy) |

**Note**: LLM metrics (`aion_llm_*`) are self-reported from the LLM response metadata. AION **does not integrate native vLLM metrics** — token counts come from the `usage` field of the API response.

### Emission via Hook

Metrics are emitted at runtime via hooks registered in `src/observability/hooks_emitter.py`:

| Hook | Emitted Metrics |
|---|---|
| `on_user_message` | `aion_messages_total` (role=user) |
| `pre_tool_use` | Timestamp registration for duration calculation |
| `post_tool_use` | `aion_tool_calls_total`, `aion_tool_call_duration_seconds` |
| `post_turn` | `aion_turn_duration_seconds`, `aion_messages_total` (role=assistant), `aion_llm_tokens_total`, `aion_llm_turn_tokens`, `aion_llm_turn_calls`, `aion_agent_failures_total`, `aion_session_cache_size_bytes` |

## Structured Logging

All core infrastructure logs use `structlog` to produce structured logs (JSON or human-readable text). The configuration is managed by `src/observability/logging.py`.

### Formatting

The format is selectable via `AION_LOG_FORMAT`:

| Value | Format | Use |
|---|---|---|
| `json` (default) | JSON on stdout | Production, ingestion in Loki/Elastic |
| `text` | ConsoleRenderer (human-readable) | Development |

### OTel Context Injection

Each log automatically includes the current `trace_id` and `span_id`, obtained from the active OTel span. Span attributes (`aion.profile`, `aion.session_id`, `tool.name`, etc.) are also injected into the logs.

The `add_otel_context` processor in `logging.py` reads the current span and adds the fields:

```json
{
  "event": "tool execution complete",
  "trace_id": "0000000000000000abcdef1234567890",
  "span_id": "0000000012345678",
  "aion.profile": "tecnico-senior",
  "aion.session_id": "sess-abc123",
  "level": "info"
}
```

### OTLP Log Export

When `AION_OTEL_ENABLED=1`, logs are also exported via OTLP using `BatchLogRecordProcessor` + `OTLPLogExporter`. This allows correlating application logs with traces in the OTel backend.

### Logging Configuration

| Variable | Default | Description |
|---|---|---|
| `AION_LOG_LEVEL` | `INFO` | Logging level (DEBUG, INFO, WARNING, ERROR) |
| `AION_LOG_FORMAT` | `json` | Format: `json` or `text` |

Logs go only to stdout (no file logging). Uvicorn and FastAPI loggers are propagated to the root logger to maintain format consistency.

## Opik — Self-Hosted LLM Telemetry

**Opik** (Comet ML) is a telemetry layer separate from the OTel pipeline, dedicated specifically to LLM tracking, prompt library and session feedback scores. It is designed to be self-hosted and communicate with an Opik instance on an internal network.

### Opik Architecture

```
┌─────────────────────────────────┐
│         Agent Runtime            │
│                                  │
│  ┌──────────────┐  ┌──────────┐  │
│  │ opik_setup.py│  │opik_telemetry.py│
│  │ (SDK init,   │  │ (@track  │  │
│  │ Prompt Lib)  │  │ decorators)│  │
│  └──────┬───────┘  └────┬─────┘  │
│         │               │         │
│         └───────┬───────┘         │
│                 │                 │
└─────────────────┼─────────────────┘
                  │ HTTP
                  ▼
┌─────────────────────────────────┐
│      Opik (Comet ML)            │
│                                 │
│  • Prompt Library               │
│  • Traces (LLM, tool, skill)    │
│  • Threads (multi-turn)         │
│  • Feedback Scores              │
└─────────────────────────────────┘
```

### Setup (`src/observability/opik_setup.py`)

The `opik_setup.py` module manages the initialization of the Opik SDK:

1. Checks if `AION_OPIK_ENABLED` is set to `1` (or `true` / `yes` / `on`). If not enabled, it completely bypasses Opik client initialization and sets `OPIK_AVAILABLE` to `False`.
2. Reads `OPIK_URL_OVERRIDE` and `OPIK_PROJECT_NAME` from environment variables.
3. Automatically sets a placeholder for `OPIK_API_KEY` if not configured (required for client validation in self-hosted mode).
4. Initializes the Opik client with `Opik()`.
5. Exports `OPIK_AVAILABLE` (bool) for graceful degradation.

If `AION_OPIK_ENABLED=0`, the SDK is not installed, or the server is unreachable, `OPIK_AVAILABLE` becomes `False` and no requests fail — the application continues normally without attempting connections.

### Prompt Library

`get_or_create_prompt()` manages the versioning of prompt templates:

1. Searches for the prompt by name in the Opik Prompt Library
2. If it exists and the template has changed → creates a new version (commit)
3. If it exists and the template is unchanged → returns the existing prompt
4. If it does not exist → creates a new prompt with the provided template

The template uses Jinja-style placeholders (e.g. `{{profilo}}`, `{{user_input}}`) that are dynamically formatted at runtime via `prompt_obj.format()`.

### Tracking Decorators (`src/observability/opik_telemetry.py`)

The module provides `@track` decorators to mark functions as Opik spans of different types:

| Decorator | Type | Function | Description |
|---|---|---|---|
| `@track(type="llm")` | `call_llm_api()` | Real LLM call with token tracking, temperature, endpoint |
| `@track(type="tool")` | `execute_tool_ricerca()` | Tool execution with input/output in the span |
| `@track(type="general")` | `skill_ricerca_db()`, `skill_calcolo_metrico()` | Agent skills as nested spans |
| `@track(type="general")` | `run_agent()` | Main root trace of the agent |

### Agent Integration

Opik wrappers are applied at runtime in:

- **`src/main.py:1169-1261`**: Opik wrappers applied to `chat_generator` and MCP tools
- **`src/agent_pipeline.py:987-1015`**: Opik trace initialization in the agent pipeline

### Session Feedback

`chiudi_sessione_e_valuta()` sends session-level feedback scores to Opik at the end of the interaction. Currently it sends a fixed score of `1.0` for the `risoluzione_problema` metric, but the design supports future extensions for real feedback (e.g. user thumbs up/down).

### OTel vs Opik Distinction

| Aspect | OTel | Opik |
|---|---|---|
| Purpose | Generic infrastructure | LLM-specific |
| Traces | Generic spans (HTTP, DB, tool) | LLM, prompt, skill spans |
| Metrics | Prometheus + OTel push | No metrics |
| Prompt Library | None | Yes, with versioning |
| Feedback Score | None | Yes, at session level |
| Backend | Phoenix, Jaeger, Tempo, etc. | Opik (Comet ML) |

### Opik Configuration

| Variable | Default | Description |
|---|---|---|
| `OPIK_API_KEY` | `local-self-hosted-placeholder` | Opik API key (placeholder if not set) |
| `OPIK_PROJECT_NAME` | `AION-Agent` | Opik project name |
| `OPIK_URL_OVERRIDE` | `http://localhost:5173/api` | API endpoint of the self-hosted Opik instance |

### Per-step LLM call audit (local JSON)

Separate from Opik/OTel: when `AION_LLM_CALL_AUDIT=1`, `src/runtime/llm_call_audit.py` writes full request/response payloads for each agent LLM step to `data/diagnostics/llm_calls/` (override with `AION_LLM_CALL_AUDIT_DIR`). CLI: `python scripts/audit_llm_calls.py`. Intended for dev/staging regression on tool-call JSON and vLLM thinking behavior — not a replacement for production trace export.
